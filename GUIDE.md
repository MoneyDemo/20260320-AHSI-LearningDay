GitHub Copilot: # Phase 1：GitHub Spark 的 Prompt 架構

## Prompt 的四大要素（SCIB 框架）

給學員一個好記的框架，讓他們撰寫 Spark prompt 時有結構可循：

---

### **S — Structure（結構）**
> 告訴 Spark「頁面長什麼樣」

- 有哪些頁面 / 區塊？
- 排版方式是什麼？（卡片、時間軸、列表、網格）
- 資訊的層次關係？（摘要 → 展開詳情）

```
❌ 模糊：「幫我做一個旅行網站」
✅ 具體：「做一個單頁網站，上方有行程總覽導航列，
         下方用垂直時間軸呈現每天的行程，
         每天是一張卡片，裡面有多個景點」
```

---

### **C — Content（內容）**
> 告訴 Spark「裡面放什麼資料」

- 提供具體的資料範例（不要只說「放一些景點」）
- 每個項目要有哪些欄位？（名稱、時間、地址、費用…）
- 資料量大概多少？（5天、每天3-5個景點）

```
❌ 模糊：「每天放幾個景點」
✅ 具體：「Day 1 包含：
         - 淺草寺（10:00-12:00，免費，東京都台東區淺草2-3-1）
         - 仲見世通（12:00-13:30，午餐預算 ¥1500）
         - 晴空塔（15:00-17:00，門票 ¥2100）」
```

---

### **I — Interaction（互動）**
> 告訴 Spark「使用者可以做什麼」

- 點擊會發生什麼？（展開、跳轉、彈窗）
- 有沒有狀態變化？（勾選完成、收藏、評分）
- 資料會不會被修改？（新增備註、調整順序）

```
❌ 模糊：「要可以互動」
✅ 具體：「點擊景點卡片可以展開看到詳細資訊，
         每個景點有一個 checkbox 可以標記已完成，
         完成的景點卡片變成灰色並移到底部」
```

---

### **B — Brand（風格）**
> 告訴 Spark「整體視覺感受」

- 配色方向（不需要給色碼，給氛圍就好）
- 字體感覺（現代、手寫、日式）
- 參考風格（某個你知道的網站風格）

```
❌ 模糊：「好看一點」
✅ 具體：「日式簡約風格，白色底色搭配淡紅色點綴，
         卡片有圓角和淡陰影，
         標題用較粗的無襯線字體，
         整體感覺像 Notion 的乾淨排版」
```

## 實際演練：漸進式 Prompt

教學員用 **3 輪對話** 建出網站，而非一次塞完所有需求：

| 輪次 | 策略 | 範例 |
|------|------|------|
| **Round 1** | 先給 S + C，建立骨架 | 「做一個 5 天東京旅行時間軸，每天有景點卡片，包含名稱和時間」 |
| **Round 2** | 加 I，加入互動 | 「讓卡片可以點擊展開，顯示地址和費用，加上完成勾選功能」 |
| **Round 3** | 調 B，微調風格 | 「改成日式配色，白底紅色點綴，卡片加圓角陰影」 |

### 為什麼要分輪？

> 這是 Workshop 的**重要教學點**：
> - Spark 跟 ChatGPT 一樣，**一次要求太多容易失焦**
> - 分輪迭代可以**確認每一步的產出**再往下走
> - 如果某一輪結果不對，只需要**重講那一輪**，不用全部重來
> - 這也是實際軟體開發的 **iterative 思維**

---

## Prompt 反模式（常見錯誤）

| 反模式 | 問題 | 改正 |
|--------|------|------|
| 🚫 太抽象 | 「做一個很酷的旅行網站」 | 沒有任何具體指引，結果不可預測 |
| 🚫 太技術 | 「用 React + Tailwind + Framer Motion」 | Spark 自己決定技術實作，不需要指定框架 |
| 🚫 一次全塞 | 把 20 個功能寫在一個 prompt 裡 | 分輪迭代，每次聚焦 2-3 個改動 |
| 🚫 只說不要 | 「不要太醜、不要太擠」 | 正面描述你要什麼，而非不要什麼 |

GitHub Copilot: # GitHub Spark Prompt 策略大全

## 策略一：角色扮演法（Persona）

> 讓 Spark 扮演某個角色來產出

```
你是一位專業的日本旅行規劃師，
幫我設計一個 5 天東京旅行的互動行程網站。
網站要像你會交給客戶的那種精緻行程表，
包含每日重點、交通指引、費用估算。
```

**適合時機：** 你不確定細節該怎麼安排，讓 Spark 用「專家思維」幫你補完

---

## 策略二：模仿參考法（Reference）

> 描述一個你見過的產品，讓 Spark 照著做

```
做一個類似 Google Trips 風格的旅行行程頁面：
- 頂部有目的地的大圖 banner
- 下方是每天行程的可折疊區塊
- 每個景點有小圖示+簡短描述
- 底部有打包清單 checklist
整體感覺乾淨、Material Design 風格
```

**適合時機：** 你腦中有明確的畫面，但不想從零描述每個細節

---

## 策略三：使用者故事法（User Story）

> 從「誰在用、怎麼用」出發

```
我是一個即將去東京自由行的旅客。
我希望在手機上打開這個網站，
可以快速看到「今天要去哪裡」，
點擊景點看到怎麼搭車過去，
去過的地方可以打勾，
這樣我不用一直翻旅行書。
```

**適合時機：** 功能導向，讓 Spark 從使用情境反推該做什麼 UI

---

## 策略四：漸進增量法（Incremental）

> 每次只加一個功能，逐步堆疊

```
Round 1: 「列出 5 天東京行程，每天 3-4 個景點，用卡片排列」
Round 2: 「每張卡片加上可以展開的詳細資訊區塊」
Round 3: 「加上每個景點之間的交通方式和步行時間」
Round 4: 「頂部加一個 Day 1-5 的 tab 切換」
Round 5: 「每個景點加上 checkbox，勾選後變灰色」
```

**適合時機：** 最穩健的方式，每一步都可以確認結果再往下

---

## 策略五：約束驅動法（Constraints First）

> 先告訴 Spark 限制條件，再說功能

```
限制條件：
- 必須在手機上好用（響應式設計）
- 不需要登入功能
- 所有資料寫死在頁面裡（不需要後端）
- 頁面載入要快，不用大圖片

在這些條件下，做一個 5 天東京旅行行程網站，
用時間軸呈現，可以點擊展開景點詳情。
```

**適合時機：** 你很清楚「不要什麼」，用限制來縮小 Spark 的發揮範圍

---

## 策略六：Before/After 對比法

> 告訴 Spark 現在的痛點，讓它解決

```
我現在的旅行行程是記在 Google Doc 裡，問題是：
- 看起來就是一堆文字，沒有視覺層次
- 不知道景點之間怎麼移動
- 沒辦法標記已完成的地方
- 手機上看很痛苦

幫我把這份行程變成一個好用的互動網站：
[貼上你的 Google Doc 行程內容]
```

**適合時機：** 你已經有原始資料，只是要 Spark 幫你「升級呈現方式」

---

## 策略七：情緒板法（Mood Board）

> 用形容詞和氛圍描述，而非技術規格

```
我想要一個讓人看了就想出發旅行的行程網站。

氣氛：清新、令人期待、像翻開一本旅行雜誌
色調：櫻花粉 + 天空藍 + 大量留白
感覺：每滑一段就像發現一個新驚喜
圖示：用可愛的 emoji 代替制式 icon

內容是 5 天東京行程，每天 3-5 個景點。
```

**適合時機：** 你在意的是「感覺」而非功能，適合視覺導向的學員
