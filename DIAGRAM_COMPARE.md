| 優先 | 視覺化 | 給**工程師**的益處 | 給 **EM / Tech Lead** 的益處 | 給 **PM / 利害關係人**的益處 | ADO/PBI 完全做不到的獨特價值 |
|:----:|--------|-------------------|---------------------------|---------------------------|--------------------------|
| 1st | **Interactive Mind Map** | 一眼看到自己的 Task 在整棵樹的哪個位置，理解上下文 | 發現不平衡的分支（某 Feature 下堆了太多 Task，另一個卻空的）；識別缺少 Owner 的孤兒節點 | 不需讀 Backlog 列表就能理解整個 Epic 的工作分解全貌；截圖放進簡報秒懂 | ADO Backlog 是「縮排列表」，永遠看不到樹的**形狀** — 哪條分支深、哪條淺、哪裡密集、哪裡稀疏 |
| 2nd | **Animated Timeline** | 看到自己的 Task 是何時被建立的、距離上游 Epic 建立已過多久，感受到交付壓力的時間脈絡 | 回顧 Sprint 時用「播放」重現工作演進，精準指出「哪一天活動暴增」「哪段時間完全停滯」 | 用 30 秒動畫向高階主管「說故事」— 觀看工作如何從 Epic 一路往下展開到 Task，比任何靜態報表都更有說服力 | ADO/PBI 零動畫能力；CFD 只看「量的堆疊」看不到「事件的因果先後」與「狀態變更的波紋效應」 |
| 3rd | **Force-Directed Graph** | 發現自己的工作跟哪些 Sibling Task 有關、跟哪些人有交集，主動溝通協作 | 識別隱性的協作群集（誰跟誰常在同一棵樹上工作）；發現被過度指派的人（節點連線過多） | 理解團隊結構與工作結構是否對齊 — 跨團隊的關聯是否有人負責協調 | ADO 完全沒有「網路圖」概念；Sibling 關係在 Backlog 視圖中不可見，只有 Parent-Child 的縮排 |
| 4th | **Treemap** | Task 的 Remaining/Completed Work 雙色進度條，比 ADO 的百分比數字更直覺地看到「還剩多少」 | 用面積立刻看出哪個 Feature 佔了最多 Effort，資源分配是否合理；顏色一眼區分 New（灰）vs Active（藍）的比例 | 一張截圖就能回答老闆的問題：「我們的時間都花在哪裡？」— 面積大小 = 投入量，比圓餅圖更能呈現 4 層巢狀 | PBI Treemap 最多 2 層、不支援 Parent-Child 自動解析、無法在格子內嵌入 RW/CW 進度條 |
| 5th | **Calendar Heatmap** | 一眼看到自己在哪些天最忙（Created + Updated 密度），回顧自己的工作節奏 | 發現「沉默的週」（整週零事件 = 項目可能卡住）和「爆發的日」（一天 10+ 事件 = 可能是趕死線）；用於 Sprint Retro 的客觀數據 | GitHub-style 的日曆格最符合直覺 — 不需解讀圖表，顏色深淺 = 忙碌程度，hover 就看到當天發生什麼事 | ADO 無日曆視圖（只有 Sprint 日期範圍）；PBI 需用 Matrix + 條件格式拼湊，無法做到 hover 顯示事件明細 |

### 5 個視覺化的互補關係

```
Mind Map (結構) ──→ "這棵樹長什麼樣？"
     │
     ▼
Animated Timeline (時間) ──→ "這棵樹怎麼長出來的？"
     │
     ▼
Force Graph (關係) ──→ "樹上的人和節點怎麼連結？"
     │
     ▼
Treemap (資源) ──→ "每根樹枝佔了多少資源？"
     │
     ▼
Calendar Heatmap (節奏) ──→ "哪些日子在澆水施肥？"
```

五張圖合在一起，等於用**結構 → 時間 → 關係 → 資源 → 節奏**五個維度完整剖析一棵工作項目樹。
